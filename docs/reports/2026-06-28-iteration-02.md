# Iteration 02 — Modelo profissional estruturado: Technology

## Objetivo

Implementar a primeira entidade do modelo profissional estruturado do ProfileSync AI, permitindo registrar tecnologias e competências vinculadas a perfis profissionais.

Esta iteração marca a transição do sistema de uma aplicação centrada apenas em currículos para uma plataforma baseada em ativos estruturados de conhecimento profissional.

## Escopo implementado

- Criação da entidade `Technology`.
- Relacionamento entre `Technology` e `Profile`.
- Criação dos schemas de entrada, atualização e resposta.
- Implementação da camada de persistência.
- Implementação da camada de serviço.
- Implementação das rotas HTTP autenticadas.
- Registro das rotas no `main.py`.
- Testes automatizados para repository, service e API.
- Validação de controle de acesso por usuário.

## Funcionalidades entregues

### Technology Model

Implementada a entidade `TechnologyModel`, representando tecnologias e competências associadas a um perfil profissional.

Campos principais:

- `id`
- `profile_id`
- `name`
- `category`
- `proficiency_level`
- `years_experience`
- `created_at`

### Technology Repository

Implementadas operações de persistência:

- criar tecnologia;
- listar tecnologias por perfil;
- buscar tecnologia por ID;
- atualizar tecnologia;
- excluir tecnologia.

### Technology Service

Implementadas regras de negócio com validação de propriedade por usuário.

O serviço garante que tecnologias só possam ser criadas, listadas, consultadas, atualizadas ou removidas quando vinculadas a um perfil pertencente ao usuário autenticado.

### Technology API

Criadas rotas autenticadas para gerenciamento de tecnologias:

- `POST /technologies/profiles/{profile_id}`
- `GET /technologies/profiles/{profile_id}`
- `GET /technologies/{technology_id}`
- `PUT /technologies/{technology_id}`
- `DELETE /technologies/{technology_id}`

## Decisões arquiteturais

A entidade `Technology` foi utilizada também como representação inicial de competências técnicas e profissionais.

A separação entre `Technology` e `Skill` foi evitada neste momento para reduzir duplicidade conceitual e manter o domínio simples, extensível e adequado à fase atual do projeto.

O modelo adotado permite representar diferentes tipos de competência por meio do campo `category`, como:

- linguagem;
- framework;
- banco de dados;
- DevOps;
- metodologia;
- compliance;
- soft skill.
